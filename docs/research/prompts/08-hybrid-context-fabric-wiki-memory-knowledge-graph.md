# Deep Research Prompt: Hybrid Context Fabric — Wiki, Memory, Knowledge Graph & Project Context Packs

## Purpose

Research best practices for designing a hybrid context and memory system for AI software engineering agents. The system should not be a manually maintained wiki — it should automatically ingest, structure, index, retrieve, evaluate, and govern knowledge from multiple sources.

## When to Run

Run after Topics 01, 02, and 05 (Priority 1, tied with core architecture topics). The context fabric is foundational to Conductor's self-improvement capability and agent effectiveness.

## Deep Research Prompt

Copy the following into ChatGPT Deep Research:

---

**Research best practices for designing a hybrid context and memory system for AI software engineering agents.**

**The system should not be a manually maintained wiki. It should automatically ingest, structure, index, retrieve, evaluate, and govern knowledge from multiple sources.**

Context:
- I'm building a product orchestration platform (called "Conductor") that uses AI agents (Claude Code) to build software products.
- A file-based wiki repo currently stores ingested company context, Confluence docs, repo information, client information, ticket information, deployment context, and other operational entities. The user does not want to manually maintain this wiki.
- A separate memory system (zeus-memory) may provide semantic memory/vector retrieval and reusable memory patterns. It should be treated as an optional provider, not a hard dependency.
- Conductor should become the context router, governance layer, project memory manager, and context pack builder.
- Each project/product/client should have its own scoped memory and generated knowledge workspace.
- The platform is multi-tenant — memory must respect tenant/project/client boundaries.

I need to understand:

1. **Hybrid retrieval architecture** — how to combine source documents, vector DB, metadata filters, structured stores, and knowledge graphs into a unified retrieval system. How to decide which retrieval method to use for a given query. How to merge results from different sources.

2. **Source-of-truth separation** — how to separate original source documents (wiki pages, tickets, code, Confluence) from generated/derived memory (syntheses, learned patterns, agent observations). How to maintain provenance. How to handle contradictions between sources and derived memory.

3. **Project-level knowledge workspaces** — how to build per-project/per-product knowledge workspaces that are automatically generated and maintained without manual wiki curation. What a workspace contains (entities, relationships, decisions, risks, context packs, learned patterns). How workspaces evolve over a project's lifecycle.

4. **Source adapters** — how to design pluggable adapters for ingesting knowledge from: wiki repos (markdown), ticket systems (Jira/Linear), code repositories (git log, file structure, README), documentation (Confluence, Google Docs), research outputs (markdown reports), and memory systems (vector DB, semantic stores). Common adapter interfaces and data normalization patterns.

5. **Metadata-first filtering** — how to use structured metadata (tenant_id, project_id, source_type, freshness, confidence, evaluation_status) to filter results BEFORE running expensive semantic/vector search. How this improves retrieval quality and respects tenant isolation. Index design for fast metadata filtering.

6. **Tenant/client/project isolation** — how to implement knowledge isolation across tenants, clients, and projects. How to share global knowledge (patterns, best practices) without leaking tenant-specific data. How to handle knowledge that spans multiple projects for the same client.

7. **Stale and conflicting knowledge detection** — how to detect when knowledge items are outdated (the source changed), contradicted (two sources disagree), superseded (a newer decision overrides an older one), or low-confidence (generated by an uncertain process). How to surface conflicts for human review.

8. **Context packs for AI agents** — how to build scoped context bundles that give an AI agent exactly the knowledge it needs for a specific task. Pack types: project context pack, phase context pack, review context pack, deployment context pack. How to size packs to fit context windows. How to rank and prioritize items within a pack.

9. **Memory evaluation and self-improvement** — how to evaluate whether memory items are helping or hurting agent performance. Metrics for memory quality. Feedback loops from build outcomes. How to promote good memories, quarantine bad ones, and roll back changes that degrade performance.

10. **Memory promotion, quarantine, and rollback** — lifecycle for memory items: candidate → evaluated → trusted → global. What gates each transition. How to quarantine without losing context. How to roll back a batch of promoted memories if they degrade performance.

11. **UI for knowledge management** — how to expose the knowledge system through a UI for: semantic search, knowledge graph browsing, memory approval/rejection, source inspection and provenance, context pack review and customization, conflict resolution, freshness/staleness dashboards.

Provide your output as a structured report with:
- Recommended architecture (with component diagram in text form)
- Data model (entities, relationships, metadata schema)
- Source adapter design (interfaces, normalization)
- Retrieval pipeline (metadata filter → vector search → rerank → assemble)
- Knowledge graph strategy (entity types, relationship types, graph queries)
- Vector memory strategy (embedding model, chunk strategy, index design)
- Project memory strategy (workspace structure, lifecycle, generation)
- Evaluation and self-improvement loop (metrics, feedback, promotion/quarantine)
- UI views (search, graph, approvals, packs, conflicts)
- Risks and mitigations
- MVP implementation plan (what to build first)
- Advanced roadmap (full system)

Reference existing systems: LangChain, LlamaIndex, Mem0, Zep, MemGPT, Notion AI, Microsoft Graph RAG, knowledge graph-enhanced RAG papers. Compare approaches.
